# Todo-List PyQt6 桌面应用 —— 从零构建学习报告

---

## 一、指导思想：先搭骨架，再填血肉

构建一个桌面应用，最忌讳的是一上来就写界面代码。正确的顺序是：

```
数据层 → 逻辑层 → UI 层 → 美化 → 打包
```

如果把软件比作盖房子：

- **数据层** = 地基 — 数据存在哪里？长什么样？
- **逻辑层** = 水电管道 — 怎么读写数据？怎么流转？
- **UI 层** = 墙壁装修 — 用户看见什么？怎么交互？
- **美化** = 软装 — 颜色、间距、圆角
- **打包** = 交房 — 让别人能直接用

本项目严格遵循这个顺序，每一步都是可测试的、独立的。


**核心原则：单一职责。** 每个 `.py` 文件只负责一个清晰的任务：
- `models.py` 只管数据长什么样
- `database.py` 只管怎么存取数据
- `main_window.py` 只管主窗口的布局和交互
- `style.qss` 只管视觉样式

这样做的好处是：修改数据库逻辑时不会碰坏 UI，调整颜色时不会弄乱数据。

## 三、数据层：先把数据想清楚

### 3.1 用 dataclass 定义数据结构

[models.py](todo_app/models.py) 是整个项目最简单的文件，但也是最重要的文件——它定义了"任务"长什么样：

```python
from dataclasses import dataclass
from typing import Optional

@dataclass
class Task:
    id: Optional[int] = None          # 数据库自增ID，新建时为None
    title: str = ""                   # 标题
    description: str = ""             # 描述
    priority: str = "medium"          # high / medium / low
    due_date: Optional[str] = None    # "2026-05-21" 或 None
    status: str = "pending"           # pending / completed
    created_at: str = ""              # 创建时间
    completed_at: Optional[str] = None # 完成时间
```

**为什么用 dataclass 而不是普通 dict？**
- 有 IDE 自动补全和类型检查
- 不会把 `titel` 拼错但运行时才发现
- `from_row()` 工厂方法统一了"从数据库行 → Python 对象"的转换

### 3.2 SQLite 数据库设计

[database.py](todo_app/database.py) 的建表语句：

```sql
CREATE TABLE IF NOT EXISTS tasks (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    title       TEXT NOT NULL,
    description TEXT NOT NULL DEFAULT '',
    priority    TEXT NOT NULL DEFAULT 'medium'
                CHECK(priority IN ('high','medium','low')),
    due_date    TEXT,          -- ISO 日期字符串
    status      TEXT NOT NULL DEFAULT 'pending'
                CHECK(status IN ('pending','completed')),
    created_at  TEXT NOT NULL DEFAULT (datetime('now','localtime')),
    completed_at TEXT
);
```

**几个细节值得注意：**

1. **`IF NOT EXISTS`** — 程序每次启动都会执行建表语句，但只建一次。这保证了"首次运行自动建库"。
2. **`CHECK` 约束** — 防止写出 `priority = "urgent"` 这种非法值。数据库层面的最后防线。
3. **`TEXT` 存日期** — SQLite 没有 DATE 类型，用 TEXT 存 ISO 格式字符串（`"2026-05-21"`）是最通用的做法。
4. **`DEFAULT (datetime('now','localtime'))`** — 让数据库自动填入创建时间，不需要 Python 代码操心。

**数据库文件放哪？**

```python
# utils.py
def get_data_dir():
    if getattr(sys, 'frozen', False):
        base = os.environ["APPDATA"]           # 打包后：%APPDATA%/TodoListPyQt6/
    else:
        base = os.path.dirname(...)             # 开发中：项目根目录/
    path = os.path.join(base, "TodoListPyQt6")
    os.makedirs(path, exist_ok=True)           # 目录不存在就创建
    return path
```

**`sys.frozen`** 是 PyInstaller 打包后设置的标志——这是判断"开发环境 vs 打包后"的标准方法。

### 3.3 参数化查询与防 SQL 注入

注意所有 SQL 操作都使用 `?` 占位符：

```python
cursor = self._conn.execute(
    "SELECT * FROM tasks WHERE id=?", (task_id,)   # ✅ 安全
)
```

**永远不要用 f-string 拼 SQL：**
```python
# ❌ 危险！用户输入 "1; DROP TABLE tasks--" 会导致删表
f"SELECT * FROM tasks WHERE id={user_input}"
```

### 3.4 动态构建查询

`get_all_tasks()` 展示了动态 WHERE 子句的标准写法：

```python
def get_all_tasks(self, filter_params: FilterParams = None) -> list[Task]:
    sql = "SELECT * FROM tasks WHERE 1=1"   # 基石：永远为真
    params = []

    if filter_params.search_text:
        sql += " AND (title LIKE ? OR description LIKE ?)"
        params.extend([f"%{filter_params.search_text}%", f"%{filter_params.search_text}%"])

    if filter_params.priority != "All":
        sql += " AND priority = ?"
        params.append(filter_params.priority.lower())
    # ... 更多条件

    cursor = self._conn.execute(sql, params)
    return [Task.from_row(row) for row in cursor.fetchall()]
```

**`WHERE 1=1`** 是一个经典技巧：
- 它永远为真，不改变查询结果
- 后续所有条件都可以统一加 `AND ...`，不用判断"这是第几个条件，要不要加 WHERE"

## 四、UI 层：Qt 的核心思想

### 4.1 Qt 编程的基本范式

PyQt6 程序的运行模型是一个**事件循环**：

```
创建 QApplication → 创建窗口 → 连接信号 → 显示窗口 → app.exec() 进入循环
```

在 `app.exec()` 之后，程序就进入了一个无限循环，不断等待和响应用户的操作（点击、键盘、鼠标等）。每一个操作都称为一个**事件**。

### 4.2 信号与槽（Signal & Slot）

这是 Qt 最核心的机制，也是**理解 Qt 的关键**：

```python
# 信号：用户点击按钮
self._clear_filter_btn.clicked.connect(self._clear_filters)
#                        ↑                    ↑
#                      信号                  槽（回调函数）
```

你可以把信号和槽理解为"当 A 发生时，自动调用 B"：
- **信号(signal)** = 发生了什么事（clicked、textChanged、triggered...）
- **槽(slot)** = 发生后要执行的函数

**为什么不用直接在按钮上绑回调？** 因为信号槽是类型安全的、支持多对多连接、跨线程安全。

本项目 MainWindow 的所有信号连接在 [main_window.py:178-200](todo_app/ui/main_window.py#L178)：

```python
def _connect_signals(self):
    # 过滤器：文字改变 → 刷新列表
    self._search_box.textChanged.connect(self._apply_filters)

    # 表格：双击 → 编辑任务
    self._table.doubleClicked.connect(self._on_edit_current_task)

    # 菜单：菜单项点击 → 对应操作
    self._act_add.triggered.connect(self._on_add_task)
```

### 4.3 QMainWindow 的结构

`QMainWindow` 是一个"完整的主窗口"，它预置了以下区域：

```
┌──────────────────────────────────────────┐
│ QMenuBar   (文件 | 视图 | 帮助)           │  ← menuBar()
├──────────────────────────────────────────┤
│ QToolBar   (➕ 添加 | ✓ 完成 | 🗑 删除)   │  ← addToolBar()
├──────────────────────────────────────────┤
│                                          │
│ setCentralWidget(QWidget)                │  ← 你放什么就是什么
│   ├── Filter Bar  (搜索/优先级/状态)     │
│   └── QTableView  (任务列表)             │
│                                          │
├──────────────────────────────────────────┤
│ QStatusBar (共 N 个任务 | X 待完成 | ...) │  ← statusBar()
└──────────────────────────────────────────┘
```

**Central Widget** 是你自己构造的 QWidget，里面用布局管理器（QVBoxLayout / QHBoxLayout）安排子控件。

### 4.4 布局管理器 vs 绝对定位

```python
# ❌ 绝对定位：窗口一拖就乱
label.move(140, 130)

# ✅ 布局管理器：自动适应窗口大小
layout = QVBoxLayout()
layout.addWidget(search_box)
layout.addWidget(table_view)
central_widget.setLayout(layout)
```

Qt 的布局系统会自动处理控件的排列、伸缩和对齐。永远用布局，不要手写坐标。

### 4.5 QAbstractTableModel：Model/View 分离

这是本项目最精妙的部分。传统的做法是直接操作 `QTableWidget`：

```python
# ❌ 传统做法：数据和 UI 耦合
table.setItem(0, 0, QTableWidgetItem("买咖啡"))
```

本项目的做法：

```
数据库 (SQLite)
    ↕  DatabaseManager
Task 对象列表 (list[Task])
    ↕  TaskTableModel (QAbstractTableModel)
QTableView (用户看见的表格)
```

**TaskTableModel** 是中间的翻译官。Qt 的 TableView 不直接接触数据，它只问 Model：

- TableView："第 2 行第 1 列显示什么？" → Model 返回 `task.title`
- TableView："第 2 行用什么颜色？" → Model 返回红色（高优先级）

核心代码在 [task_table_model.py:30-60](todo_app/ui/task_table_model.py#L30) 的 `data()` 方法：

```python
def data(self, index, role=Qt.ItemDataRole.DisplayRole):
    task = self._tasks[index.row()]
    col = index.column()

    if role == Qt.ItemDataRole.DisplayRole:      # "显示什么文字？"
        if col == 0: return task.title
        ...
    if role == Qt.ItemDataRole.ForegroundRole:   # "什么颜色？"
        if col == 1: return QColor("#C0392B")    # 高优先级=砖红
        ...
```

**这样做的好处：**
- 数据更新后只需调用 `self.layoutChanged.emit()`，表格自动刷新
- 颜色、对齐、提示文字全部由 Model 控制，不在 UI 代码里散落
- Model 可以单独测试，不需要打开窗口

### 4.6 Modal 弹窗的正确用法

[task_dialog.py](todo_app/ui/task_dialog.py) 展示了"添加/编辑"弹窗的标准模式：

```python
class TaskDialog(QDialog):
    def __init__(self, parent=None, task=None):
        self.result_task = None          # 结果通过这个属性返回

    def _on_accept(self):
        # 校验...
        self.result_task = Task(title=..., ...)  # 组装结果
        self.accept()                            # 关闭弹窗（Accepted）

# 调用方：
dlg = TaskDialog(self, existing_task)            # 如果是编辑，传已有 task
if dlg.exec() == QDialog.DialogCode.Accepted:    # exec() 阻塞等待
    db.update_task(dlg.result_task)              # 读取结果
```

关键点：
- **同一个类处理添加和编辑**：构造时传了 task 就是编辑模式，不传就是添加模式
- **结果从属性获取**：`dialog.result_task`，而不是全局变量
- **`exec()` 是阻塞的**：弹窗关闭前，主窗口代码暂停执行

### 4.7 系统托盘集成

[tray_manager.py](todo_app/ui/tray_manager.py) 展示了托盘的核心逻辑：

```python
class TrayManager(QObject):
    show_requested = pyqtSignal()       # 自定义信号
    add_task_requested = pyqtSignal()
    exit_requested = pyqtSignal()
```

**为什么用自定义信号而不是直接调 MainWindow 的方法？**

因为 `TrayManager` 不应该知道 `MainWindow` 的存在——它只知道"有人需要显示窗口"，通过 `show_requested` 信号广播出去。谁接收这个信号是调用方的事。这是**解耦**的体现。

"关闭最小化到托盘"的实现：

```python
def closeEvent(self, event):
    if minimize_to_tray and self._tray.is_available():
        self.hide()           # 不真正关闭，只是隐藏
        event.ignore()        # 告诉 Qt："别关窗口"
    else:
        event.accept()        # 正常关闭
```

## 五、美化层：QSS 样式表

### 5.1 QSS 是什么

QSS（Qt Style Sheets）的语法和 CSS 几乎一样，用来控制 Qt 控件的视觉外观：

```css
QPushButton {
    background-color: #8D6E63;    /* 焦糖色背景 */
    color: #FFFDF9;               /* 暖白文字 */
    border: none;
    border-radius: 6px;           /* 圆角 */
    padding: 7px 18px;
}
QPushButton:hover {
    background-color: #A08072;    /* 悬停时变浅 */
}
QPushButton:pressed {
    background-color: #6D4C41;    /* 按下时变深 */
}
```

和 CSS 一样，QSS 支持**层叠**和**伪状态**（`:hover`、`:pressed`、`:focus` 等）。

### 5.2 加载 QSS 的代码

```python
# __init__.py
style_path = resource_path("resources/style.qss")
with open(style_path, "r", encoding="utf-8") as f:
    app.setStyleSheet(f.read())        # 全局应用
```

一行代码，所有窗口的控件自动应用样式。不需要逐个控件设置。

### 5.3 Coffee & Cream 配色逻辑

| 色号 | 名称 | 用途 |
|------|------|------|
| `#FDFBF7` | 温润浅奶泡 | 主背景 |
| `#F3ECE0` | 燕麦拿铁 | 工具栏、表头 |
| `#3E2723` | 浓缩咖啡褐 | 主文字 |
| `#E5DEC9` | 轻柔焦糖 | 悬停/选中 |
| `#E6DFD5` | 极淡暖灰 | 边框 |
| `#8D6E63` | 深焦糖 | 按钮 |

配色原则：
- **主体用低饱和度暖色**，不刺眼
- **文字用深色保证对比度**（WCAG 可读性标准）
- **强调色只用于按钮**，引导用户视线
- **灰度变化表示状态**（悬停变浅、按下变深）

## 六、打包：从 .py 到 .exe

### 6.1 PyInstaller 做了什么

PyInstaller 把 Python 解释器、你的代码、所有依赖库、资源文件全部打包成一个 .exe：

```
你的 main.py
    +
PyQt6 (几百个 .pyd 文件)
    +
Python 解释器 (python311.dll)
    +
style.qss + app_icon.png
    ↓  PyInstaller
一个 TodoList.exe（~34MB）
```

### 6.2 关键配置

```python
# TodoList.spec
exe = EXE(
    ...,
    console=False,         # ← 关键：不显示黑色命令行窗口
    name='TodoList',
)
```

`console=False` 告诉 Windows：这是一个 GUI 程序，不要给它开 cmd 窗口。

### 6.3 打包后的资源路径问题

开发时资源路径是 `项目目录/resources/style.qss`，但打包后所有资源被解压到一个临时目录 `sys._MEIPASS`。所以需要 `resource_path()` 函数统一处理：

```python
def resource_path(relative_path):
    if getattr(sys, 'frozen', False):    # 打包后
        base_path = sys._MEIPASS
    else:                                 # 开发中
        base_path = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
    return os.path.join(base_path, relative_path)
```

**任何读取外部文件的路径都必须经过这个函数。**

## 七、注册表开机自启动

[utils.py](todo_app/utils.py) 中的 `set_autostart()` 操作 Windows 注册表：

```python
key_path = r"Software\Microsoft\Windows\CurrentVersion\Run"
key = winreg.OpenKey(winreg.HKEY_CURRENT_USER, key_path, ...)
winreg.SetValueEx(key, "TodoList", 0, winreg.REG_SZ, exe_path)
```

**为什么用 HKEY_CURRENT_USER 而不是 HKEY_LOCAL_MACHINE？**
- HKCU 不需要管理员权限
- 只会对当前用户生效
- 删掉不会影响系统

## 八、常见坑与解决方案

### 8.1 `from __future__ import annotations`

```python
# database.py 第一行
from __future__ import annotations
```

项目中使用 `Task | None` 这种语法（PEP 604，Python 3.10+），但在较老的 Python 上会报错。`from __future__ import annotations` 让所有类型注解变成惰性求值，兼容老版本。

### 8.2 应用图标的三处设置

Windows 下应用图标需要在多个地方设置：

| 位置 | 代码 | 影响 |
|------|------|------|
| 任务栏 | `app.setWindowIcon(QIcon(...))` | `__init__.py` |
| 窗口标题栏 | `window.setWindowIcon(QIcon(...))` | `main_window.py` |
| 系统托盘 | `QSystemTrayIcon(QIcon(...))` | `tray_manager.py` |
| .exe 文件图标 | 需要 .ico 格式，PyInstaller 打包时指定 | `TodoList.spec` |

**注意**：`.exe` 文件的图标和运行时窗口图标是两回事。前者需要在打包时嵌入 `.ico` 文件。

### 8.3 PyInstaller 找不到模块

PyQt6 有一些隐式导入，PyInstaller 可能检测不到。如果打包后的 .exe 启动崩溃：
1. 先用 `pyinstaller --onedir`（文件夹模式）调试，看少了哪个文件
2. 再用 `--onefile`（单文件模式）交付
3. `--onedir` 启动快，适合调试；`--onefile` 干净，适合发布

## 十、总结：构建桌面应用的思维模型

```
1. 数据模型 (dataclass)
   ↓
2. 数据库 (sqlite3 + 参数化查询)
   ↓
3. 业务逻辑 (DatabaseManager → CRUD)
   ↓
4. 数据桥梁 (QAbstractTableModel → 连接 DB 和 UI)
   ↓
5. 界面布局 (QMainWindow + 布局管理器)
   ↓
6. 交互逻辑 (Signals & Slots)
   ↓
7. 视觉美化 (QSS 样式表)
   ↓
8. 打包交付 (PyInstaller)
```

每一步都建立在前一步之上，每一步都可以独立验证。这就是从零构建一个桌面应用的全过程。
